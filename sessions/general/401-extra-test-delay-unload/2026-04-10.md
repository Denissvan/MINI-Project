## Observed Symptom

- `401` is a special add-on test executed around unload position.
- After `401` final result returns, the station should unload directly.
- In field logs, the station often does not unload immediately and instead starts another test round or needs one more turn.

## Current Hypothesis

- `401` is not a normal light-box intermediate state. It includes dedicated turn-to-feed behavior.
- After final `401` result returns, the C# state machine does not always convert that result into an unload-ready state in time.
- The station can remain in `TEST/WAIT`, after which the scheduler re-enters test flow instead of consuming the result for unload.

## Code Changes

- Analyzed `401` as a dedicated unload-position add-on test path.
- Tried adding a "401 result waiting for unload" protection path in C#.
- Added related logic and logs in one iteration, but field logs later suggested the running build did not fully reflect or trigger that protection.
- Confirmed from `2026-04-10.log` around `15:24:57` and `15:26:36` that the running flow hit `sta=0` after `401`, but then immediately executed `PC reset -> StartFlow` instead of marking "wait unload".
- Patched `UI/Class/WS.cs` so the `401` no-light-box / feed-position continuation path also marks `bResultWaitUnload` and `bUpDnAddTestWaitUnload` when the final `sta=0` arrives.

## Validation Result

- Root cause direction is clearer than before.
- The previous repeated-test behavior is now mapped to a specific missing branch in `WS.cs`, not just a generic timing issue.
- Local source has been patched, but site build has not yet been field-verified.

## Update 2026-05-06 22:08:55

- Field snapshot confirmed the direct cause: after `401` final `sta=0,n=8`, `myfunc.cs` unload decision saw `TestStatus=NEXT`, `FeedStatus=REDAYFORTEST`, `readyByCompleted=False`, `readyByFeedStatus=False`, then entered the `else` branch and started the same barcode batch again.
- Applied a minimal 401-only closure in `UI/Class/WS.cs`: when final `sta=0` returns while `bUpDnPosGoOnTest` is active, set `Status=REDAY`, `TestStatus=COMPLETED`, clear `bUpDnPosGoOnTest`, set `bUpDnAddTestWaitUnload=true`, and log `401追加测试最终结果已返回，等待下料消费`.
- Updated `UI/Class/myfunc.cs` unload decision so `bUpDnAddTestWaitUnload` is treated as unload-ready, logs it in `下料判定快照`, clears it after entering the unload path, and blocks repeated `StartFlow` if the flag is still pending.
- Corrected the guard to avoid using outer `num` as result count. The 401 closure now checks `sta == 0 && bUpDnPosGoOnTest`.
- This patch is intentionally narrow: it does not change the generic DLL result protocol, 403/Hall flow, GRR/点检 logic, or normal non-401 unload criteria.
