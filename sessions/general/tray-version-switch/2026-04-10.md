## Observed Symptom

- Startup can choose different machine versions.
- In swapped tray mode, NG materials may physically go to the OK tray array.
- Run-page display and actual unload target can become inconsistent.
- In swapped tray mode, some code paths still follow fixed physical OK/NG axes, so startup version selection does not fully switch the working axes.

## Current Hypothesis

- The old implementation mixed logical tray identity, physical tray mapping, and UI display labels.
- Some code paths judged OK/NG by `traybox.disc` display text instead of a stable role.
- Run UI also performed an extra swap, which could disagree with action-layer mapping.

## Code Changes

- Added stable tray logical role abstraction for feed/OK/NG.
- Adjusted tray creation logic so version switch changes physical mapping without changing logical identity.
- Reworked action-side OK/NG judgment away from display text.
- Reworked run-page binding to reduce double-swap risk.
- Fixed constructor ambiguity compile issue after introducing tray role support.
- Removed swapped-mode second-pass UI exchange in `FrRun`, so run-page tray display always follows logical OK/NG ownership.
- Changed reset-page tray axis status display in `FrRst` to read `COM.traybox_ok/ng.ax_x/ax_z` instead of fixed `MT.AXIS_UDL_OK/NG_*`.
- Changed unload tray Z-axis move protection in `MT` to compare against the actual traybox object's mapped X/Z axes, so swapped mode protects and executes against the currently bound physical axis pair.
- Added tray startup tracing in `COM.TrayBoxInit()` to log startup mode and each traybox logical-to-physical axis/config mapping during initialization.
