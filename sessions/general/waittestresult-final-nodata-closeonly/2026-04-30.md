## Observed Symptom

- Startup with multiple stations enabled entered first-cycle close/reset logic.
- Main log stopped at `工站1 WaitTestResult,ID=1`.
- DLL repeatedly printed `WaitTestResult FinalNoData,id=1,result_mark=4,status=0,param=...`.
- Result area was `mark=4,sta=0,num=0,res=[-1,...]`.

## Root Cause

- `NextTest(-1)` is a close-only/reset command and does not represent a real product test.
- The later `WaitTestResult()` saw close-only empty result `MARK_END/status=0/num=0`.
- Previous `FinalNoData` protection kept waiting for real test results, but did not distinguish close-only empty result from real StartFlow pending result.
- The `FinalNoData` branch used `continue` before the loop update, so the timeout countdown was not consumed on that path.

## Code Changes

- `dllforcomv8/myfunc.c`
  - Added `g_closeOnlyWaitPending[256]`.
  - `NextTest(status == -1)` sets close-only pending only after the command is acknowledged.
  - `NextTest(status == -1)` also marks close-only pending when the result area is already `MARK_END` and the command is skipped as completed.
  - `StartTestFlow()` clears close-only pending before handling a real start command.
  - `WaitTestResultA()` consumes close-only empty result `MARK_END/status=0/num=0` and returns `RES_OK` with log `WaitTestResult CloseOnlyNoDataOk`.
  - Real StartFlow `status=0,num=0` still waits for valid result data.
  - `FinalNoData` now decrements the remaining delay before `continue`, preventing infinite wait.

## Expected Log Signature

- For first-cycle close/reset:
  - `NextTest CloseOnly`
  - `NextTest Acked`
  - `WaitTestResult CloseOnlyNoDataOk`
- For real test with delayed result:
  - `WaitTestResult FinalNoData` may appear briefly.
  - It must either receive `num>0` and return normally, or exit by timeout; it must not loop forever.
