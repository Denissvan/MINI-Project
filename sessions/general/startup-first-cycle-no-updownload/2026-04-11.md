## Observed Symptom

- After machine reset/startup, the very first cycle starts test logic before any real updownload/load has completed.
- Field behavior then degrades into "no updownload" or direct `NOBARCODE` / duplicate-barcode startup flow.

## Key Evidence

- Main log shows `09:16:01.865` created updownload thread, but `09:16:01.875` already started test thread.
- WS logs immediately execute first-cycle `NextTest(-1)` / `WaitTestResult`, before real load state is established.
- At `09:16:23.938`, the scheduler still drives `е·Ґз«™4 еђЇеЉЁжµ‹иЇ•`, and later `PC4/PC8 StartFlow` carries `NOBARCODE` payloads.
- Test-side DLL log confirms early `ResetTest` / `NextTest(-1)` / `InfArea(... barcode=EMPTY)` behavior in the same startup window.

## Current Hypothesis

- The existing first-cycle bootstrap in `WS.RunTest()` resets test software and then marks the station as completed, but does not force one real updownload before allowing normal test scheduling.
- If startup is interrupted or the state is not yet physically established, the scheduler can re-enter `StartFlow` with synthetic/empty material state.

## Code Changes

- Added `bStartupInitPending` in `WS` to mark "startup initialization completed but real updownload not done yet".
- After the first-cycle reset branch, the station now enters that pending state instead of freely flowing into normal test scheduling.
- In `myfunc.cs`, when this pending state is present, the station is forced to `REDAYFORUPDOWNLOAD` before any direct test restart.
- In `UpDownLoad.cs`, that pending flag is cleared only after one real updownload completes.
