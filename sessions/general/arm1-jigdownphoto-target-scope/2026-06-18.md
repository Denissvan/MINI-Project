# Arm1 JigDownPhoto Target Scope

- Log analyzed: `C:\Users\gdfanyd\Downloads\2026-06-17.log`
- Decoding: GBK.
- Symptom: user observed arm 1 only scans/checks fixture 3,4,7,8 and not 1,2,5,6.
- Key settings in log:
  - `bitOpenMode: 1`
  - `bJigSan: False`
  - `OpenDownQrde: True`
  - `UpDownRunMode` mostly `0` during production, meaning `BOTH_WORK`.
- Code mapping:
  - `myfunc.GetRunMode`: `BOTH_WORK` maps `UDLoad1.FindMod = HIGH`, `UDLoad2.FindMod = LOW`.
  - `UpDownLoad.GetWsPosTeam`: `HIGH` skips `n % 8 < 4`, so module 1 takes the high half of each 8 positions.
  - `WS.MdDat.benable`/save logic: `bitOpenMode == 1` disables even `Num`.
  - Therefore UD1/HIGH + single-open leaves internal `Num=5,7,13,15`, mapping to physical fixtures `3,4,7,8`.
- Log evidence:
  - UD1 JigDownPhoto cached targets dominated by `5,13`, `5,13,7,15`, `5`, `5,7`.
  - UD2 JigDownPhoto cached targets dominated by `1,9`, `1,9,3,11`, which maps to physical fixtures `1,2,5,6`.
- Current conclusion:
  - The observed target split is caused by run-mode allocation plus single-open filtering, not by random missing scan.
  - `bJigSan=False`, so the log is not showing the dedicated fixture barcode scan path; it is showing down/up-camera QR/check and JigDownPhoto target caching.

## 2026-06-18 Code Change

- File changed: `UI/Class/UpDownLoad.cs`
- Goal: decouple `WsModAfterClose_Shp.vpp` execution from the arm that placed the material.
- Behavior after change:
  - Placement remains split by current run mode.
  - After fixture close, the code merges `UDLoad1` and `UDLoad2` after-close targets.
  - In dual-arm mode, arm 1 is selected as the unified executor if any arm-1 up camera has `WsModAfterClose_Shp.vpp`.
  - If arm 1 has no such VPP but arm 2 does, arm 2 executes the unified target list.
  - In single-arm modes, the configured arm is preferred, with fallback to the other arm only if that is where the VPP is loaded.
  - Per target, the executor first uses the expected camera for the module side if it has the VPP, otherwise it uses any executor camera that has the VPP.
  - Targets placed by the other arm are converted to the executor arm's workstation coordinates before triggering the VPP.
