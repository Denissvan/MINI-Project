# Left Lightbox Delayed After Updownload

- Date: 2026-04-23
- Workshop: general / not yet specified
- Demand track: `left-lightbox-delayed-after-updownload`

## Symptom

After the updownload-position test is completed, station 3 does not start the left-lightbox test in the expected cycle. It starts only in the next round.

## Evidence

- `19:33:09.711` PC3 `StartFlow,ret=成功,sta=0`
- `19:33:09.712` station 3 `通知REDAYFORTEST`
- `19:33:10.538` station 3 `WaitTestResult,ID=3`
- `19:33:10.539` station 3 `PC3,sta=8,n=0`
- `19:33:10.539` station 3 `WaitTestResult,ID=7`
- `19:33:10.539` station 3 `PC7,sta=8,n=0`
- `19:34:20.714` station 3 `转下一位置...`
- `19:34:23.856` `切换工位周期`

## Code Context

- `UI/Class/myfunc.cs`: scheduler starts test flow after updownload and runs Hall pre-processing.
- `UI/Class/WS.cs`: station test thread maps positions to light boxes: `POS1=LeftLightBox`, `POS2=OTPLightBox`, `POS3=RightLightBox`.
- `dllforcomv8/myfunc.c`: `WaitTestResultA` returns for both final and intermediate statuses when result mark is ready/end.

## Current Analysis

- Station 3 receives `sta=8,n=0` immediately after `StartFlow`.
- In the Hall pre-processing branch, only `sta == 301` is forwarded by `NextTest(sta)`.
- Any non-301 status enters `else { break; }`, so `sta=8` is not acknowledged/forwarded there.
- Since `n=0`, this is not a final product result. It is an intermediate requested test position.
- The updownload-position pre-processing exits with the first lightbox-position request still pending. The real lightbox move/`NextTest(8)` is therefore deferred until the station thread runs at the next applicable position/cycle.

## Next Action

Consider changing the Hall pre-processing logic so non-final intermediate statuses are not silently broken out:

- Keep special handling for `301`.
- Treat `sta == 0` as final result.
- For other positive intermediate statuses such as `8`, either defer explicitly with trace logs or execute the same lightbox-position handling path used by `WS.RunTest`.
- Add logs around `sta != 301 && sta != 0` to prove whether the first requested left-lightbox step is being delayed.
